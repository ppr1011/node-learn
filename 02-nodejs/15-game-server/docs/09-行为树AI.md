敌人 AI 一开始是手写状态机。随着「巡逻 / 追击 / 攻击 / 逃跑 / 狂暴」越加越多，`if-else` 会迅速缠成一团。本部分记录把敌人 AI 重构为**行为树**的过程——一个真实、跑得起来的案例，而非教科书玩具。

## 1. 为什么从状态机换成行为树

重构前的 `EnemyAISystem.tickEnemy` 是典型的 FSM：

```ts
if (target) {
  if (dist <= attackRange) { aiState = 'attack'; tryAttack(); }
  else { aiState = 'chase'; moveToward(target); }
} else {
  patrol();
}
```

问题不在于它跑不动，而在于**加行为的成本**：想让史莱姆残血逃跑、恶魔残血狂暴，就得往这团 if-else 里塞更多分支和标志位，条件互相耦合，越改越脆。

**行为树（BT）** 把「决策」变成一棵可组合的树。每 tick 从根节点向下遍历，用两个组合节点表达几乎所有控制流：

| 概念 | 状态机（FSM） | 行为树（BT） |
|------|--------------|--------------|
| 基本单元 | 状态 + 迁移边 | 节点(组合/装饰/叶子) |
| 加一个行为 | 加状态 + 维护所有相关迁移 | 插一个子树分支 |
| 优先级 | 靠迁移条件手写顺序 | Selector 从左到右天然表达 |
| 「进行中」 | 靠状态停留 | `running` 状态一等公民 |
| 复用 | 状态逻辑难跨敌人复用 | 叶子/子树可跨树复用 |

## 2. 三种执行结果

BT 的核心语义是每个节点 tick 后返回三选一（见 `src/ai/bt/types.ts`）：

- **success**：完成且成功（「已攻击」「已到达」）
- **failure**：无法执行（「视野里没有目标」）
- **running**：仍在进行，下一 tick 继续（「正在追击/巡逻」）

组合节点据此决定走向——这就是全部魔法。

## 3. 通用节点库

`src/ai/bt/nodes.ts` 只有 5 个与游戏无关、可复用的节点：

```ts
Selector  // fallback:依次 tick,遇到第一个「非 failure」即返回;全 failure 才 failure
Sequence  // 依次 tick,遇到第一个「非 success」即返回;全 success 才 success
Inverter  // 装饰:success↔failure 取反(running 透传)
Condition // 叶子:判定 → success/failure
Action    // 叶子:执行 → success/failure/running
```

配套 `sel() / seq() / cond() / act()` 简写工厂，让树的声明读起来接近伪代码。

**节点无状态**：所有「记忆」（巡逻点、狂暴标志、攻击冷却）都存在 `enemy` 字段上，通过黑板 `BTContext` 访问。因此**同一种敌人共享一棵树实例**，成千上万个敌人零额外内存。

## 4. 敌人的树

`src/ai/bt/enemyTree.ts`：

```
Selector(                          // 有目标就战斗,否则巡逻
  Sequence(
    Condition: 探测到目标? ────────── 写入黑板 ctx.target
    Selector(                        // 锁定后的优先级决策
      Sequence(残血 & 是史莱姆 → 逃跑),   // 1. slime 怂
      Sequence(在攻击距离内 → 攻击),       // 2. 够得着就打(内部冷却门控)
      追击                                 // 3. 够不着就追(demon 残血自动狂暴)
    )
  ),
  巡逻                              // 无目标:回出生点附近游荡(running)
)
```

读法是「从上到下、从左到右的优先级」：能锁定目标就进战斗子树，否则退回巡逻；锁定后，先判断要不要逃，再判断能不能打，都不行就追。

叶子实现都在 `src/ai/bt/enemyActions.ts`（逻辑基本是从旧 FSM 平移过来的）：动作只设置 `enemy.velocity` 和 `enemy.aiState`（客户端据此显示 chase/attack 光环），真正的位移 + 边界收敛由 `EnemyAISystem` 在 tick 末尾统一 `applyMovement`。

## 5. 种类差异：同一棵树，不同性格

树按 `kind` 构建（`buildEnemyTree(kind)`），差异通过闭包捕获 `kind` 注入到条件里：

- **史莱姆（slime）**：残血分支命中 `kind === 'slime' && isLowHp` → 触发 `flee`，朝反方向加速逃离；其它种类这个 `Condition` 恒 `failure`，自然落到攻击/追击。
- **恶魔（demon）**：`chase` 动作内检测 `demon 残血` → 置 `enraged = true`，速度 ×1.4 且不再复原（复活时在 `Enemy.respawn` 重置）。
- **骷髅（skeleton）**：不进这两个特殊分支，就是标准的探测→追击→攻击。

新增性格只是「换一个条件谓词」或「插一个分支」，互不干扰。

## 6. 执行器只剩三行

重构后 `EnemyAISystem` 退化成纯执行器：

```ts
if (enemy.isDead) { /* 处理复活计时 */ continue; }
const ctx = { enemy, world, dt, now, target: null };
this.treeFor(enemy.kind).tick(ctx);   // 决策全在树里
this.applyMovement(enemy, dt);        // 落地位移
```

「怎么决策」与「怎么执行」彻底分离。

## 7. 扩展思考

- **呼叫增援 / 绕后 / 群体逃跑**：再插一个 `Sequence` 分支即可，无需动其它逻辑。
- **装饰器扩展**：可加 `Cooldown`（限制某分支的触发频率）、`Repeat`、`Succeeder` 等装饰节点，进一步减少叶子里的手写计时。
- **数据驱动**：树结构可以进一步从 JSON/可视化编辑器加载，策划无需改代码就能调 AI——这正是 BT 在商业引擎（Unreal Behavior Tree、Unity 各类 BT 插件）里成为标配的原因。
- **LLM 战术层**：大模型不适合每帧控制，但适合做高层意图决策；见 [11 · LLM + 行为树 AI](11-LLM行为树.md)。
