### 1. 实时状态同步

游戏服务器与 Web 后台最大的区别：**状态是连续变化的**。

Web 后台是请求-响应模型，客户端发一个请求、服务器返一个响应。而游戏服务器需要持续广播世界状态给所有玩家，每秒 20+ 次。

```
传统 Web：  Client → Request → Server → Response → Client（一次性）
游戏服务器：Client → Input → Server → Broadcast → All Clients（持续循环）
```

**服务器权威模式**（本 Demo 采用）：

```typescript
// 客户端只发送意图（"我要向右走"），不直接修改坐标
// 服务器负责校验和计算最终位置
handleInput(player, data) {
  const { dx, dy } = data;
  // 归一化 + 速度限制，防止客户端发送非法速度
  const len = Math.sqrt(dx * dx + dy * dy);
  player.velocity = {
    x: (dx / len) * player.speed,  // 服务器决定最终速度
    y: (dy / len) * player.speed,
  };
}
```

**为什么不让客户端直接同步坐标？**
- 客户端可以被篡改（外挂），发送虚假坐标
- 多个客户端对同一状态的修改可能冲突
- 服务器无法做碰撞检测和规则校验

### 2. AOI (Area of Interest) — 兴趣区域管理

**问题**：假设有 1000 个玩家在线，每人每秒移动产生 20 次状态更新。如果每次都广播给所有人：`1000 × 1000 × 20 = 2000 万次/秒`，网络根本扛不住。

**解决方案：九宫格 AOI**

将地图划分为固定大小的格子（本 Demo 为 500×500 像素）。每个玩家只需要知道自己所在格子及相邻 8 个格子（九宫格）内的玩家动态。

```
┌─────┬─────┬─────┬─────┬─────┐
│     │     │     │     │     │
├─────┼─────┼─────┼─────┼─────┤
│     │ NW  │  N  │ NE  │     │
├─────┼─────┼─────┼─────┼─────┤
│     │  W  │ [P] │  E  │     │   P 只关心周围 9 格
├─────┼─────┼─────┼─────┼─────┤
│     │ SW  │  S  │ SE  │     │
├─────┼─────┼─────┼─────┼─────┤
│     │     │     │     │     │
└─────┴─────┴─────┴─────┴─────┘
```

**关键实现**：当玩家跨越格子边界时，计算新旧九宫格的差集，得到 enter/leave 事件。

```typescript
updatePlayer(player) {
  const [newCx, newCy] = this.getCellCoords(player.position.x, player.position.y);
  if (newCx === player.aoiCellX && newCy === player.aoiCellY) return; // 没跨格

  // 计算差集
  const oldNeighbors = this.getNeighborPlayers(oldCx, oldCy);
  const newNeighbors = this.getNeighborPlayers(newCx, newCy);
  // entered = newNeighbors - oldNeighbors
  // left    = oldNeighbors - newNeighbors
}
```

**效果**：广播量从 O(N²) 降到 O(N × K)，K 是单格平均玩家数。

### 3. 游戏主循环 (Game Loop / Tick)

传统 Web 服务器是**事件驱动**的：有请求才处理。游戏服务器需要**主动推进时间**：即使没有玩家操作，世界也在运转（怪物巡逻、BUFF 倒计时、物理模拟等）。

```typescript
// 固定频率 Tick（本 Demo 20Hz = 每 50ms 一次）
class GameTimer {
  start() {
    this.timer = setInterval(() => {
      const elapsed = Date.now() - this.lastTime;
      this.accumulator += elapsed;

      // 固定步长更新，保证物理模拟稳定
      while (this.accumulator >= intervalMs) {
        this.onTick(intervalMs);  // 固定 50ms 的 deltaTime
        this.accumulator -= intervalMs;
      }

      // 防止螺旋死亡：如果积累太多帧，直接丢弃
      if (this.accumulator > intervalMs * 5) {
        this.accumulator = 0;
      }
    }, intervalMs);
  }
}
```

**为什么不用 `setInterval(tick, 50)` 直接跑？**

`setInterval` 不保证精确间隔。如果某一 tick 耗时过长（比如 80ms），下一个 tick 会延迟，导致物理模拟不稳定。`accumulator` 模式确保逻辑更新步长恒定。

**螺旋死亡（Spiral of Death）**：当 tick 处理时间 > tick 间隔时间，accumulator 不断增长，每次需要补更多帧，越补越多。解决方案：设置上限，超过就直接丢弃。

### 4. 协议设计

本 Demo 使用 JSON 格式方便学习，但生产环境需要考虑：

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| JSON | 可读、调试方便 | 体积大、序列化慢 | 开发/低频消息 |
| Protocol Buffers | 小体积、强类型、跨语言 | 需要编译 .proto 文件 | 生产环境首选 |
| FlatBuffers | 零拷贝、极快 | 使用复杂 | 超高频消息 |
| 自定义二进制 | 最小体积 | 维护困难 | 特殊场景 |

**消息结构设计原则**：
```typescript
// 好的设计：消息类型 + 数据分离
{ type: "c_move", data: { dx: 1, dy: 0 } }

// 坏的设计：每种消息不同结构
{ action: "move", x: 1, y: 0, player: "abc" }
```

### 5. 数据一致性

多个玩家同时操作同一目标时，可能产生竞态条件：

```
玩家A 攻击怪物（HP=10, 伤害=8）→ HP 应该 = 2
玩家B 攻击怪物（HP=10, 伤害=8）→ HP 应该 = 2
实际结果应该是 HP = -6（死亡），而非两个独立的 HP=2
```

**解决方案**：
- 单线程模型（Node.js 天然优势）：所有逻辑在同一线程执行，无需加锁
- Tick 内顺序执行：同一 tick 内的操作按顺序处理，保证一致
- 对于持久化数据：使用乐观锁 + 版本号

### 6. 空间碰撞检测（障碍物）

移动不能只做边界 clamp——世界里有障碍物，玩家不能穿墙。碰撞系统同样遵循**服务器权威**：世界里所有**实心物体（树、石）都由服务端生成、通过 `JOIN_WORLD` 下发**，碰撞解算只在服务端进行，客户端仅负责按下发数据渲染。灌木、花草等纯装饰物才由客户端本地生成、不参与碰撞。

> ⚠️ **一致性要点**：不要让「看起来实心」的东西没有碰撞。如果树/石只在客户端用随机种子各画各的、服务端毫不知情，就会出现「一棵树挡在面前却能直接穿过去」的割裂感。把实心物体收归服务端权威，是让「所见 = 所判定」的关键。

**① 圆形碰撞体（按类型给不同半径）**

障碍物和玩家都用圆表示（`{ x, y, radius }`）。相比多边形/AABB，圆的优势是：

- 碰撞判定只需一次距离比较：`distance(A, B) < rA + rB` 即相交
- 旋转无关，无需维护朝向
- 推出方向天然就是两圆心连线方向

对每 tick 高频调用的服务端来说，这种「便宜」至关重要。不同类型的障碍物**碰撞半径与视觉尺寸解耦**：

| 类型 | 碰撞半径 | 渲染 | 图层 |
|------|----------|------|------|
| 树 (tree) | 树干（`size × 0.32`，远小于树冠） | 树冠大团 | 玩家**之上**（可走到树荫下被遮挡） |
| 石 (rock) | ≈ 视觉半径（整块） | 圆形巨砾 | 玩家**之下** |

树用「小树干碰撞 + 大树冠视觉 + 树冠盖在玩家头顶」，既挡住树干、又保留「玩家走到树下被遮住」的 2.5D 效果——这正好解决了「树冠浮在最上层却能穿过」的割裂。

**② 空间分割网格（broad-phase）**

如果每个玩家每帧都和全部障碍物比一遍，就是 O(玩家数 × 障碍物数)。解决思路和 AOI 完全一致——**用网格把空间切开**：

```
构建期：每个障碍物按其外接正方形覆盖的格子，登记进 Map<cellKey, Obstacle[]>
        （大障碍物会跨多格，全部登记）

查询期：玩家只查自己所在格子 + 周围 8 格（九宫格）内的候选障碍物
        O(N) 全量遍历  →  O(K) 附近少数几个
```

对应实现 `ObstacleGrid.queryNearby(x, y, radius)`，见 `src/core/Obstacle.ts`。

**③ push-out 滑动解算（narrow-phase）**

拿到候选障碍物后，如何响应碰撞？两种做法：

| 做法 | 表现 | 手感 |
|------|------|------|
| **直接停下 (block)**：撞到就把整步移动作废 | 卡死在障碍物正面，必须手动绕开 | 差 |
| **滑动 (push-out)**：沿法线推回表面，保留切向分量 | 顺着障碍物边缘滑过去 | 好 ✅ |

本 Demo 采用**滑动**。每帧先算出「期望位移」，若新位置钻进障碍物圆，就沿「障碍圆心 → 玩家」这条**法线**把玩家推回表面：

```typescript
// src/systems/MovementSystem.ts —— 圆 vs 圆 push-out
const dx = nx - o.x, dy = ny - o.y;
const minDist = player.radius + o.radius;
const dist = Math.hypot(dx, dy);
if (dist < minDist) {
  const push = minDist - dist;
  nx += (dx / dist) * push;   // 只抵消「垂直撞向障碍物」的法向分量
  ny += (dy / dist) * push;   // 「平行于表面」的切向分量被完整保留 → 滑动
}
```

法向分量被抵消、切向分量保留，玩家就沿边缘滑过去而不是卡死。对**夹在两个障碍物之间**的情况，做 2 遍解算 pass：第一遍推出一个后可能又压进另一个，第二遍再修正。

> 📊 **可视化**：项目根目录 `collision-demo.html` 用交互动画演示了「滑动 vs 停下」的差别，以及期望位移 = 法向分量 + 切向分量的向量分解。直接用浏览器打开即可，把鼠标放到障碍物正后方对比两种模式最直观。

**④ 确定性生成 + 出生点避障**

树和石都用确定性 PRNG（`Math.sin(seed)` 取小数，同种子恒定同结果）生成，配合拒绝采样按**碰撞半径 + 间隙**保证互不重叠（树因树干半径小，可以靠得近、树冠自然交叠成林）。服务端下发后所有客户端布局完全一致。玩家出生点也用 `queryNearby` 检测，拒绝采样避免生在障碍物里。

### 7. 统一物件生成模块（Spawn Module）

障碍物只是「世界里需要被生成的东西」之一。天气、敌人、掉落物……都要生成，而且各有各的随机性诉求。与其每种物件写一套散落的生成函数（早期障碍物生成就埋在 `Obstacle.ts` 里、玩家出生点又各写各的 `Math.random()`），不如抽出一个**统一生成框架** `src/spawn/`，让「新增一类物件」变成「实现一个 `SpawnDefinition` 并注册」。

**两种随机模式**是这个模块的核心区分：

| 模式 | 随机源 | 时机 | 多端一致性 | 代表物件 |
|------|--------|------|-----------|----------|
| `static`（确定性） | `seededRng(seed)`（`Math.sin` 取小数） | 启动时生成一次 | ✅ 同种子恒定同布局 | 障碍物、地形、固定物品 |
| `dynamic`（运行时随机） | `randomRng()`（`Math.random`） | 运行时/定时反复生成 | 靠**服务端权威 + 广播**保证 | 天气、刷怪、掉落 |

```
src/spawn/
├── types.ts               # Spawnable / SpatialSpawnable / SpawnContext / SpawnDefinition
├── rng.ts                 # seededRng(确定性) / randomRng(运行时) / placeMany(拒绝采样放置)
├── Spawner.ts             # 注册表:注册定义、生成、维护跨类别「占用集合」做避让
└── definitions/
    ├── obstacles.ts       # 障碍物(static)——原样迁移旧算法,布局逐像素一致
    ├── weather.ts         # 天气(dynamic,非空间全局状态)——已完整打通
    ├── enemies.ts         # 敌人(dynamic,空间)——生成骨架,默认 count=0
    └── items.ts           # 物品(static/dynamic 皆可)——生成骨架,默认 count=0
```

**关键设计**：

- **占用集合 + 跨类别避让**：`Spawner` 累积已生成的空间物件,后生成的类别通过 `ctx.occupied(x,y,r)` 自动避开先生成的（例如物品不会落进树里）。`placeMany()` 把「拒绝采样撒点」这一通用逻辑抽出来,供敌人/物品复用——和障碍物的思路一脉相承。
- **障碍物零行为变化**：迁移时原样保留旧的种子算法（`seed + seq*7 + attempt*3` 等）,新旧输出逐字段比对完全一致（55 树 + 26 石）。框架只是包了一层,`Obstacle.ts` 现在只留碰撞数据结构（`Obstacle` 接口 + `ObstacleGrid`）。
- **天气：动态 + 非空间的完整示例**：服务端每 `WEATHER_CHANGE_INTERVAL` 用 `randomRng` 重掷天气,通过 `WEATHER` 消息广播给所有玩家（`JOIN_WORLD` 也带上当前天气）。客户端按 `kind + intensity` 渲染雨/雾/雪叠加层（见 [06 · 客户端视觉系统](06-客户端视觉系统.md)）。天气用真随机,但由服务端统一决定,所以多端仍然一致——这正是「服务器权威」在非空间状态上的体现。
- **敌人/物品是可复制的扩展模板**：已注册但 `count=0` 不产出,骨架里注明了「要真正让敌人活起来」还需补的环节（实体化 → tick 更新 → 战斗接入 → 协议广播 → 客户端渲染）。
